# PRD: Генератор SQL VIEW для 1С

**Версия:** 4.5  
**Дата:** 2026-03-23  
**Статус:** Стабильная  

---

## 1. Обзор продукта

### 1.1 Проблема

Базы данных 1С используют обфусцированные технические имена таблиц (`_Document653`, `_Reference37172`) и полей (`Fld37206`, `_Fld10028RRef`), а связи между таблицами хранятся в виде GUID (`binary(16)`) без явных foreign keys. Это делает прямую работу с данными 1С через SQL крайне трудоёмкой.

Аналитики данных тратят часы на ручное построение SQL-запросов: расшифровку технических имён, поиск связей, составление JOIN-деревьев. Каждый раз при работе с новой таблицей фактов процесс повторяется заново.

### 1.2 Решение

Инструмент, который автоматически:
1. Принимает на вход таблицу фактов Ф.
2. Расшифровывает техническую структуру через файл документации (.docx).
3. Оценивает правомерность выбора Ф как таблицы фактов.
4. Строит GUID-индекс и индекс связей для всей БД.
5. Исследует окрестность Ф (связанные таблицы измерений, табличные части) через BFS/DFS.
6. Эвристически фильтрует полезные таблицы и поля.
7. Генерирует SQL-запрос для получения плоской денормализованной таблицы ФФ.

### 1.3 Целевая аудитория

**Аналитики данных**, которые:
- Знают SQL и понимают реляционную модель.
- Работают с данными 1С через BI-инструменты (например, Apache Superset).
- Нуждаются в быстром построении денормализованных витрин данных.

## 2. Пайплайн обработки

Процесс разбит на последовательные этапы. Каждый этап имеет чётко определённый вход и выход.

## 3. Функциональные требования

### Этап 0: Выбор таблицы фактов Ф

**Вход:** Действие пользователя — выбор таблицы.  
**Выход:** Имя таблицы фактов Ф (техническое имя в БД).

| ID     | Требование | Приоритет |
|--------|-----------|-----------|
| E0-01  | Приложение предоставляет **выпадающий список** всех таблиц из БД. | MUST |
| E0-02  | Список содержит как техническое, так и человеческое название таблицы (если доступно из .docx). Формат: `Человеческое (Техническое)`. | MUST |
| E0-03  | Список поддерживает **фильтрацию/поиск** по тексту (и по техническому, и по человеческому имени). | MUST |
| E0-04  | Список может быть отфильтрован по **типу таблицы**: Document, Reference, AccumRg, InfoRg, VT, Enum и др. | MUST |
| E0-05  | Пользователь выбирает **одну** таблицу фактов. Для табличных частей пользователь указывает VT-таблицу напрямую. | MUST |
| E0-06  | По умолчанию список отсортирован по типу таблицы (Document → AccumRg → InfoRg → прочие). | SHOULD |

### Этап 1: Парсинг файла структуры (.docx)

**Вход:** Файл `Структура.docx` с описанием таблиц и полей.  
**Выход:** Маппинги `техническое_имя ↔ человеческое_имя` для таблиц и полей.

| ID     | Требование | Приоритет |
|--------|-----------|-----------|
| E1-01  | Парсинг файла `.docx` извлекает: маппинг таблиц, маппинг полей, типы таблиц (Справочник, Документ, РегистрНакопления и др.). | MUST |
| E1-02  | Результат парсинга кешируется в JSON-файл (`temp/structure_parsed.json`) для ускорения повторных запусков. | MUST |
| E1-03  | Пользователь может загрузить файл структуры тремя способами: файл по умолчанию, загрузка через UI, указание пути. | MUST |
| E1-04  | Парсер корректно обрабатывает таблицы с суффиксами (основная, VT-части) и нормализует имена (добавляет подчёркивание). | MUST |

### Этап 2: GUID-индекс

**Вход:** Все таблицы 1С в БД.  
**Выход:** Словарь `{GUID (bytes) → имя_таблицы}`.

| ID     | Требование | Приоритет |
|--------|-----------|-----------|
| E2-01  | Для каждой таблицы 1С считываются значения первичных ключей (`_IDRRef`) и сохраняются в индекс. | MUST |
| E2-02  | Индекс сохраняется на диск (JSON с base64-кодированием ключей) для каждой БД отдельно. | MUST |
| E2-03  | При загрузке проверяется соответствие индекса текущему подключению (host + database). | MUST |
| E2-04  | Индекс можно перестроить принудительно через UI. | MUST |
| E2-05  | Визуализация GUID-индекса: столбчатая диаграмма распределения GUID по типам таблиц. | SHOULD |

---

### Этап 2.5: Индекс связей (relationship index)

**Вход:** GUID-индекс + все таблицы 1С.  
**Выход:** Словарь `{таблица → {поле → целевая_таблица}}`.

| ID     | Требование | Приоритет |
|--------|-----------|-----------|
| E2.5-01 | Для каждой таблицы и каждого поля `binary(16)` определяется целевая таблица через GUID-индекс (TOP 1 ненулевое значение → поиск в индексе). | MUST |
| E2.5-02 | Индекс сохраняется на диск (JSON) для каждой БД отдельно. | MUST |
| E2.5-03 | Индекс используется для **быстрого поиска обратных связей** (`find_reverse_relationships`): какие таблицы ссылаются на данную. | MUST |
| E2.5-04 | При загрузке проверяется соответствие текущему подключению. | MUST |
| E2.5-05 | Индекс можно перестроить принудительно через UI. | MUST |

---

### Этап 2.7: Статистика кардинальности полей

**Вход:** Все таблицы 1С в БД.  
**Выход:** `{таблица → {поле → {distinct_count, is_junk, data_type, max_length}}}`.

| ID     | Требование | Приоритет |
|--------|-----------|-----------|
| E2.7-01 | Для каждого поля каждой таблицы вычисляется `COUNT(DISTINCT)` по выборке `TOP(N)` строк. | MUST |
| E2.7-02 | Поля с `distinct_count ≤ junk_threshold` помечаются как мусорные (`is_junk`). | MUST |
| E2.7-03 | Статистика сохраняется на диск и загружается при повторных запусках. | MUST |
| E2.7-04 | Используется при автоматической фильтрации связей (F-01). | MUST |

### Этап 3: Оценка правомерности выбора таблицы Ф

**Вход:** Таблица Ф + данные парсинга из Этапа 1 + метаданные из БД.  
**Выход:** Оценка (балл/категория) + предупреждения с объяснениями.

| ID     | Требование | Приоритет |
|--------|-----------|-----------|
| E3-01  | Система оценивает таблицу Ф по набору эвристических признаков и выдаёт **интегральную оценку**: «Хорошо подходит», «Может подойти», «Вероятно не подходит». | MUST |
| E3-02  | Каждый сработавший признак отображается как отдельное **предупреждение с объяснением** причины. | MUST |
| E3-03  | Оценка носит **рекомендательный** характер; пользователь может продолжить работу вне зависимости от результата. | MUST |

#### Эвристические признаки оценки:

| ID     | Признак | Вес | Описание |
|--------|---------|-----|----------|
| H-01   | **Тип таблицы** | Сильный | `Document*`, `AccumRg*` → сильный положительный. `Reference*` → сильный отрицательный. `Enum` → очень сильный отрицательный. `VT` → положительный. |
| H-02   | **Наличие числовых/денежных полей** | Средний | Поля типа `numeric`, `decimal`, `float`, `money` → положительный признак. |
| H-03   | **Наличие полей-дат** | Средний | Поля типа `datetime`, `datetime2` → положительный признак. |
| H-04   | **Количество строк** | Средний | Большая таблица (> 10 000 строк) → скорее факт. Маленькая (< 100) → скорее справочник. |
| H-05   | **Наличие табличных частей (VT)** | Сильный отрицательный | Если у таблицы есть дочерние VT-таблицы → сильный отрицательный сигнал. |
| H-06   | **Количество внешних ссылок** | Слабый | Много полей `binary(16)` → скорее факт. |
| H-07   | **Ссылки на Document** | Средний | Наличие ссылок на таблицы Document через GUID-индекс. |

### Этап 4: Исследование окрестности таблицы Ф (граф связей)

**Вход:** Таблица Ф + GUID-индекс + индекс связей.  
**Выход:** Набор связанных таблиц с аннотациями (направление, тип, глубина, длина пути).

| ID     | Требование | Приоритет |
|--------|-----------|-----------|
| E4-01  | Исследование проводится в **двух направлениях**: прямые связи (Ф → измерения, `↓`) и обратные связи (другие таблицы → Ф, `↑`). | MUST |
| E4-02  | Глубина исследования **настраивается** отдельно для каждого направления. По умолчанию: ↓3 ↑1. | MUST |
| E4-03  | Обход графа выполняется алгоритмом **BFS** с отслеживанием `steps_down` и `steps_up`. | MUST |
| E4-04  | **Табличные части** (VT-таблицы) обнаруживаются и подтягиваются автоматически как обратные связи. | MUST |
| E4-05  | Для каждой связи определяется **тип**: `forward` (прямая, N:1) или `reverse` (обратная, 1:N). | MUST |
| E4-06  | Обратные связи находятся через **индекс связей** (быстрый поиск по всей БД). | MUST |
| E4-07  | Результат представляется в виде **дерева** в интерфейсе с информацией: техническое имя, человеческое имя, тип связи, уровень (ур), длина пути (дп). | MUST |
| E4-08  | Граф связей **сохраняется** на диск (JSON) и может быть загружен повторно. | MUST |
| E4-09  | При загрузке сохранённого графа проверяется актуальность (хэш параметров). | SHOULD |
| E4-10  | **Визуализация** графа связей: сетевой граф (NetworkX + Matplotlib) с цветовой кодировкой по типу таблицы. | SHOULD |

#### Фильтрация графа (постобработка DFS):

| ID     | Требование | Приоритет |
|--------|-----------|-----------|
| E4-F01 | Накопительные `steps_down` по DFS-пути от корня не превышают `max_depth_down`, а `steps_up` не превышают `max_depth_up`. Общая длина пути (дп) ≤ `max_depth_down + max_depth_up`. | MUST |
| E4-F02 | Исключаются **возвратные рёбра**: два соседних ребра между одними и теми же таблицами в противоположных направлениях. | MUST |
| E4-F03 | Исключение возвратного ребра каскадно удаляет все ветви, которые были бы доступны только через это ребро. | MUST |
| E4-F04 | Фильтрация применяется как к свежепостроенным, так и к загруженным с диска графам. | MUST |
| E4-F05 | DFS не заходит повторно в уже пройденную таблицу в рамках текущей ветки (table-path guard), предотвращая циклические обходы и потерю узлов с обратными связями от корня. | MUST |
| E4-F06 | Нормализация имён таблиц при построении дерева `_children`: связи индексируются и по оригинальному, и по нормализованному имени parent, чтобы DFS из нормализованного корня находил все связи. | MUST |

#### UI-фильтры графа:

| ID     | Требование | Приоритет |
|--------|-----------|-----------|
| E4-U01 | Фильтр по **уровню** (ур): multiselect, по умолчанию выбрано всё. Уровень = `cumulative_sd − cumulative_su` по DFS-пути от корня. | MUST |
| E4-U02 | Фильтр по **длине пути** (дп): multiselect, по умолчанию выбрано всё. Длина пути = `cumulative_sd + cumulative_su`. | MUST |
| E4-U03 | Фильтры работают по логике AND. | MUST |
| E4-U04 | При раскрытии узла отображается **полный путь** от корня: все рёбра с нумерацией, направлением (→/←), именами таблиц/полей и **накопительными** `(↓sd ↑su)` от корня до данного ребра. | MUST |
| E4-U05 | **Поиск в полях связей**: текстовый фильтр (без учёта регистра) по человеческим/техническим именам таблиц, строкам рёбер пути, человеческим именам полей. Фильтрация по узлам: если хотя бы одна связь узла совпадает — показывается весь узел. Работает по AND с остальными фильтрами. | MUST |
| E4-U06 | **Код пути**: multiselect по уникальным строкам из символов **↓** (прямая связь) и **↑** (обратная) вдоль DFS-пути от корня до связи. **Пустой выбор** — фильтр не применяется (в отличие от уровня/длины пути, где пустой выбор трактуется как «все»). Сочетание с остальными фильтрами по **AND**: после уровня и длины пути, до «только выбранное» и поиска. При смене графа выбор сбрасывается. Одинаковый код может соответствовать разным веткам (разные таблицы). | MUST |

### Этап 5: Фильтрация таблиц и полей

**Вход:** Набор таблиц и полей из Этапа 4.  
**Выход:** Отфильтрованный набор таблиц и полей, готовый для SQL-генерации.

#### Автоматические эвристики фильтрации:

| ID     | Эвристика | Действие | Переключаемая |
|--------|-----------|----------|---------------|
| F-01   | **Связи с < 2 уникальными значениями** в поле связи | Отключить связь | Да |
| F-02   | **Системные/служебные поля 1С** (`_Version`, `_Marked`, `_Predefined`, `_PredefinedID`, `_Description`, `_Code`, `_ParentIDRRef`, `_Folder`) | Логика по контексту | Да |
| F-03   | **Поля `binary(16)`** | По умолчанию исключать из SELECT; при явном выборе пользователем — включать | Нет |
| F-04   | **Пустые или почти пустые таблицы** (< 2 строк) | Отключить связь | Да |
| F-05   | **Поля типа `image`, `varbinary(max)`** (BLOB-данные) | Исключить из SELECT | Нет |

#### Фильтрация полей внутри таблиц измерений:

| Включать | Исключать |
|----------|-----------|
| `Наименование` / `_Description` | `ВерсияДанных` / `_Version` |
| `Код` / `_Code` (если информативный) | `ПометкаУдаления` / `_Marked` |
| Реквизиты с человеческими названиями | `ПредопределенныйЭлемент` / `_PredefinedID` |
| Числовые реквизиты (метрики) | `ЭтоГруппа` / `_Folder` |
| Даты (если не техническая дата) | `РодительIDRRef` / `_ParentIDRRef` |

Для **таблицы фактов Ф** включаются все содержательные поля, за исключением системных.

#### Пользовательское управление:

| ID     | Требование | Приоритет |
|--------|-----------|-----------|
| F-06   | Пользователь может **включить/отключить** любую связь (таблицу) вручную. | MUST |
| F-07   | Пользователь может **включить/отключить** отдельные поля внутри таблицы. | MUST |
| F-08   | Пользователь может переключить **тип JOIN** для каждой связи (LEFT / INNER / RIGHT). | MUST |
| F-09   | Отключение родительской связи **каскадно** отключает все дочерние связи. | MUST |
| F-10   | Переключатель **«Показывать только выбранное»** скрывает отключённые связи. | SHOULD |

### Этап 6: Генерация SQL

**Вход:** Отфильтрованный набор таблиц и полей из Этапа 5.  
**Выход:** SQL-скрипт для создания плоской денормализованной таблицы ФФ.

| ID     | Требование | Приоритет |
|--------|-----------|-----------|
| E6-01  | Генерация в **двух форматах**: `CREATE OR ALTER VIEW` и чистый `SELECT`. | MUST |
| E6-02  | **Именование полей**: два стиля — `classic` (`Поле` / `Алиас_Поле`) и `dotted` (`Таблица.Поле`). | MUST |
| E6-03  | Поля таблицы фактов Ф именуются **без префикса таблицы**. | SHOULD |
| E6-04  | **Исправление искажённых дат** 1С: `datetime2(0)` с годом ≥ 3000 → вычитание 2000 лет. Настраивается. | MUST |
| E6-05  | SQL содержит **комментарии**: заголовок с метаданными и комментарии у каждого JOIN-блока. | SHOULD |
| E6-06  | Результат доступен для **скачивания** как `.sql` файл и **копирования** из интерфейса. | MUST |
| E6-07  | Результат **сохраняется** в указанный путь на диске. | MUST |
| E6-08  | **Транзитные JOIN**: промежуточные связи, не включённые пользователем явно, но необходимые для формирования полного пути к включённой таблице, автоматически добавляются как JOIN без полей в SELECT. | MUST |

## 4. Итеративный процесс и сохранение конфигурации

| ID     | Требование | Приоритет |
|--------|-----------|-----------|
| I-01   | Интерфейс поддерживает **итеративный цикл**: изменение настроек и перегенерация SQL без повторного построения графа. | MUST |
| I-02   | Конфигурация проекта **сохраняется** в JSON: таблица Ф, настройки связей, глубина, фильтрация. | MUST |
| I-03   | Сохранённую конфигурацию можно **загрузить** для повторного использования. | MUST |
| I-04   | При загрузке конфигурации система **проверяет** актуальность (существуют ли таблицы/поля в текущей БД). | SHOULD |

## 8. Ключевые подсистемы

### 8.1 GUID-индекс

**Назначение:** Быстрое определение целевой таблицы по значению GUID из поля `binary(16)`.

- Строится путём сканирования PK-полей (`_IDRRef`) всех таблиц 1С.
- Хранится в памяти как `Dict[bytes, str]` и на диске как JSON с base64.
- Используется при построении графа связей и при оценке таблицы фактов (H-07).

### 8.2 Индекс связей (relationship index)

**Назначение:** Быстрый поиск обратных связей без тяжёлых SQL-запросов.

- Строится на основе GUID-индекса: для каждого поля `binary(16)` каждой таблицы берётся `TOP 1` ненулевое значение и ищется в GUID-индексе.
- Хранится как `Dict[str, Dict[str, str]]` — `{таблица → {поле → целевая_таблица}}`.
- Позволяет мгновенно найти все таблицы, ссылающиеся на заданную (обратные связи).

### 8.3 BFS-граф связей

**Назначение:** Построение дерева связей от таблицы фактов.

- BFS-обход с раздельным отслеживанием `steps_down` и `steps_up`.
- Прямые связи: поля `binary(16)` текущей таблицы → целевые таблицы.
- Обратные связи: другие таблицы, ссылающиеся на текущую (через индекс связей).
- Постобработка: DFS-фильтрация возвратных рёбер и ограничение глубины по накопительным sd/su.

### 8.4 Фильтрация возвратных рёбер

**Назначение:** Исключение циклических путей в графе.

- Возвратное ребро: два соседних ребра между одними и теми же таблицами в противоположных направлениях (A→B, затем B←A).
- Проверка выполняется по паре (source_table, target_table, direction), без учёта конкретного поля.
- Удаление возвратного ребра каскадно удаляет все ветви, доступные только через него.

### 8.5 Накопительные sd/su и нормализация имён

**Назначение:** Корректный расчёт уровня и длины пути по DFS-дереву.

- DFS отслеживает `cumulative_sd` (шаги вниз) и `cumulative_su` (шаги вверх) по пути от корня.
- Ребро пропускается, если `sd > max_depth_down` или `su > max_depth_up`.
- Уровень = `sd − su`, длина пути = `sd + su`.
- Подписи `(↓x ↑y)` в UI показывают накопительные значения от корня, а не BFS-значения отдельного ребра.
- `_children` индексируется и по оригинальному, и по нормализованному имени parent (с `_` и без), чтобы DFS из нормализованного корня находил все связи (включая reverse от корня).
- Table-path guard предотвращает повторный заход в уже пройденную таблицу по текущей ветке DFS.

### 8.6 Транзитные JOIN и визуализация конфигурации

**Назначение (SQL):** Включение промежуточных связей в SQL для формирования полного пути.

- Если пользователь включил таблицу на глубине 2, но не включил промежуточную таблицу на глубине 1, система автоматически добавляет промежуточную связь как «транзитный» JOIN.
- Транзитные JOIN генерируют `JOIN` в SQL, но **не добавляют** поля в `SELECT`.

**Визуализация конфигурации (кнопка в секции 11)** — **уже**, чем полный набор JOIN в SQL:

- В граф попадают **только** узлы (таблицы), у которых есть **хотя бы одно выбранное поле**, и **транзитные** таблицы на **пути от корня** к таким таблицам (рёбра — объединение путей `gen_rel_key_to_path` для включённых связей с выбранными полями; без подмешивания всех связей графа).
- Связь «включена», но все поля сняты и узел не лежит на пути к таблице с полями — **не показывается**.
- Набор рёбер в картинке может отличаться от полного списка JOIN в SQL (SQL по-прежнему строится по правилам генератора VIEW).

### 8.7 Обратные связи (reverse) — JOIN и описание

**Назначение:** Корректная генерация JOIN и подсчёт полей для обратных связей (VT, регистры, другие таблицы, ссылающиеся на целевую).

- **Прямая связь (forward):** `ON [родитель].[field_name] = [потомок].[pk]` — FK в родителе, PK в потомке.
- **Обратная связь (reverse):** `ON [родитель].[pk] = [потомок].[field_name]` — PK в родителе, FK в потомке (поле связи в таблице, ссылающейся на целевую).
- При сохранении конфигурации и в описании «Таблицы и выбранные поля» для обратных связей используется **source** (таблица-источник), а не target.

### 8.8 Дефолтное состояние полей

**Назначение:** Согласованное начальное состояние при открытии формы или сбросе конфигурации.

- **Дефолт:** все поля не выбраны (все чекбоксы сняты).
- Применяется при: «Новая конфигурация (сброс)», построении графа, загрузке графа из «Сохранённые графы».
- При загрузке сохранённой конфигурации или восстановлении из last_session — восстанавливается сохранённое состояние.

## 9. Границы и ограничения

| Ограничение | Описание |
|-------------|----------|
| Только 1C | Инструмент рассчитан только на базы данных 1С в формате MS SQL Server. |
| Один факт | Одно представление строится на основе одной таблицы фактов. |
| Без M:N | Связи типа «многие ко многим» не поддерживаются. |
| Без агрегаций | Генерируется плоская таблица без GROUP BY / агрегатных функций. |
| Без инкрементальности | Каждая генерация создаёт полное представление. |
| Эвристический подход | Определение связей основано на эвристиках (GUID-индекс, TOP 1), а не на гарантированных FK. |

## 10. Критерии успеха

1. Аналитик может получить готовый SQL VIEW за **< 5 минут** (вместо часов ручной работы).
2. Сгенерированный SQL **корректно выполняется** на целевой БД без ручных правок.
3. Имена полей в результирующей таблице **понятны** бизнес-пользователям.
4. Эвристики фильтрации **корректно** отсеивают > 80% нерелевантных полей без ручного вмешательства.
5. Конфигурация **переиспользуема**: сохранённый проект можно загрузить и получить тот же результат.
6. Индекс связей обеспечивает **полный** поиск обратных связей по всей БД за доли секунды.
7. Граф связей корректно исключает возвратные рёбра и соблюдает ограничения глубины по sd/su.
8. Уровни узлов корректно включают отрицательные значения (обратные связи от корня).
9. Транзитные JOIN автоматически включаются для формирования полных путей к включённым таблицам.
